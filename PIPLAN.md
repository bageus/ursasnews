# PiPlan: Ursas News (внутренний инструмент)

## 1) Цель проекта
Создать **внутренний** веб-инструмент с максимально простым интерфейсом для:
1. Сбора и отбора новостей из RSS/парсеров + AI-поиска.
2. Программной генерации видеороликов «медвежьих новостей» по сценариям из готовых ассетов.
3. Управления рубриками (индексы, топы, число дня, проекты и т.д.).

---

## 2) Принципы
- Минималистичный UI: вкладки, таблицы, формы, без сложной анимации в админке.
- Надежность пайплайнов важнее «красоты».
- Воспроизводимость рендера: один сценарий = одинаковый результат.
- Модульность: источники, LLM, TTS/«рев-движок», рендер и рубрики развиваются независимо.

---

## 3) Основные вкладки интерфейса

### Вкладка A — «Лента новостей»
**Задачи:** сбор, фильтрация, маркировка и подготовка новостей к сценарию.

Функции:
- Подключение источников:
  - RSS-ленты.
  - Веб-парсеры (по селекторам/адаптерам).
  - API-источники (если появятся).
- Нормализация новости:
  - `title`, `summary`, `content`, `source`, `url`, `published_at`, `language`, `tags`.
- Фильтры и маркеры:
  - Белые/черные списки слов.
  - Маркеры типа: `рынок`, `крипта`, `акции`, `экосистема ursas`, `медведи (животные)`.
- AI-поиск и AI-переформулирование:
  - Поле запроса «найди новости по теме…».
  - Кнопка «сжать в тезисы для сценария».
- Подключение моделей:
  - Выбор провайдера LLM (локальная/внешняя).
  - Шаблоны промптов для классификации/суммаризации.
- Очередь «в выпуск»:
  - Ручной выбор новостей, изменение порядка, блокировка дублей.

### Вкладка B — «Генерация ролика»
**Задачи:** программная сборка видео из сценария и ассетов.

Функции:
- Редактор сценария (JSON + визуальная форма):
  - Сцены/блоки выпуска.
  - Текст реплик ведущего.
  - Управление субтитрами.
- Конструктор выпуска из готовых секций:
  - Интро («Всем привет, с вами Ursas News»).
  - Новости по очереди.
  - Рубрики (индекс, проекты, топ падений и т.д.).
  - Аутро.
- Параметры экспорта:
  - Разрешение (1080x1920, 1920x1080, 1080x1080).
  - FPS, кодек, формат файла.
- Кнопки:
  - `Preview` (черновой рендер).
  - `Render final`.
  - `Export package` (сценарий + метаданные).

### Вкладка C — «Рубрики»
**Задачи:** централизованная логика постоянных сегментов.

Базовые рубрики:
- Ursas Index (сигнал «насколько медвежий рынок»).
- Ursas Projects (апдейты экосистемы).
- Top падений (монеты/акции/другое).
- Число дня.
- Ursas Tube / игровые топы.
- Новости из мира реальных медведей:
  - Если свежих новостей нет — брать из пула «резервных».
  - Использованная резервная новость помечается как «уже показана» и повторно не выдается.

---

## 4) Сценическая модель видео (MVP)
Для первого этапа сцена фиксированная:
- Студия, ведущий-медведь за столом.
- Анимации: рот (lip-sync), иногда руки, легкий поворот головы.
- На задней доске показывается текущая новость.
- Далее эта же новость показывается fullscreen карточкой.
- Нижняя строка субтитров — опционально.

### Поведение по таймлайну (одна новость)
1. Появление заголовка на доске.
2. Реплика ведущего (рев-озвучка + субтитры).
3. Переход на fullscreen карточку новости.
4. Короткий комментарий.
5. Возврат в студию.

---

## 5) Озвучка: «язык рева»
Задача — не просто TTS, а консистентное звучание ключевых фраз.

Решение:
- Словарь устойчивых фраз (`phrase_bank`):
  - Пример: «Всем привет, с вами Ursas News».
  - Каждой фразе соответствует заранее утвержденный аудио-паттерн.
- Генератор фраз из «токенов рева»:
  - Новые тексты собираются из фонем/слогов рева.
- Кэширование:
  - Один и тот же текст -> тот же seed/паттерн -> одинаковое звучание.

---

## 6) Ассеты: стратегия хранения
Рекомендуется гибридная схема:
- **В коде/репозитории:** только легкие обязательные шаблоны и конфиги.
- **Во внешнем локальном хранилище (NAS/S3/minio):** тяжелые ассеты (PSD/PNG-сеты, аудио, фоны, спрайты).

Почему:
- Репозиторий остается легким.
- Версионирование ассетов через manifest + checksum.
- Проще обновлять арт-пак без релиза кода.

Минимум для реализации:
- `asset_manifest.json` с путями, версиями, контрольными суммами.
- Проверка целостности ассетов перед рендером.

## 7) Предлагаемая архитектура

### Frontend (внутренний кабинет)
- Простое SPA (например, React + tab layout).
- Основные экраны:
  - FeedDashboard
  - VideoBuilder
  - RubricsCenter
  - Settings (LLM, парсеры, ассеты, рендер)

### Backend API
- Auth (внутренняя SSO или токен).
- Новости:
  - ingest / normalize / deduplicate / classify.
- Сценарии:
  - CRUD сценариев и шаблонов.
- Рендер:
  - постановка задач в очередь, статус, логи.
- Рубрики:
  - расчет индексов, выбор топов, fallback-логика.

### Worker-слой
- `collector-worker` (RSS/парсеры).
- `nlp-worker` (LLM-классификация/саммари).
- `voice-worker` (сборка рев-озвучки).
- `video-render-worker` (ffmpeg/рендер-пайплайн).

### Хранилища
- PostgreSQL: метаданные, сценарии, рубрики, статусы.
- Redis/Queue: фоновые задачи.
- Object storage: ассеты и готовые ролики.

## 8) Модель данных (упрощенно)

### NewsItem
- `id`, `source_id`, `title`, `summary`, `content`, `url`, `published_at`
- `category`, `sentiment`, `score_bearish`, `tags[]`
- `is_duplicate`, `is_selected`

### EpisodeScript
- `id`, `date`, `status`, `target_format`
- `segments[]` (news/rubric/intro/outro)
- `subtitle_enabled`, `voice_profile`

### RubricState
- `date`, `ursas_index`, `number_of_day`
- `top_fallers[]`, `projects_updates[]`
- `bear_world_news_primary`, `bear_world_news_fallback_used`

### AssetManifest
- `asset_id`, `type`, `version`, `storage_uri`, `checksum`, `is_active`

## 9) PiPlan по этапам

## Этап 0 — Discovery (3–5 дней)
- Формализовать список источников новостей.
- Утвердить рубрики и правила приоритизации.
- Зафиксировать формат сценария (JSON schema).
- Утвердить минимальный набор ассетов для MVP.

**Выход:** технический бриф и схема данных v1.

## Этап 1 — MVP News Feed (1–2 недели)
- Подключить 3–5 RSS источников.
- Реализовать нормализацию и удаление дублей.
- Вкладка ленты: фильтры, маркеры, ручной отбор в выпуск.
- AI-поле поиска/суммаризации (одна модель).

**Выход:** редактор может сформировать пул новостей на выпуск.

## Этап 2 — MVP Video Builder (2–3 недели)
- Вкладка генерации ролика со сценарием.
- Фиксированная сцена (студия + доска + fullscreen карточка).
- Базовые анимации медведя из ассетов.
- Экспорт 1 формата (например, 1080x1920).

**Выход:** программная сборка первого рабочего ролика.

## Этап 3 — Rubrics Engine (1–2 недели)
- Вкладка «Рубрики».
- Ursas Index + число дня + top падений.
- Логика fallback для «новостей реальных медведей» без повторов.

**Выход:** автоматическое заполнение постоянных секций.

## Этап 4 — Voice Engine v1 (1–2 недели)
- Реализовать phrase bank фиксированных фраз.
- Реализовать генератор «рева» для новых текстов.
- Привязать к субтитрам и lip-sync таймкодам.

**Выход:** консистентная «фирменная» озвучка.

## Этап 5 — Stabilization (1 неделя)
- Логи, мониторинг, retry задач.
- Проверка качества выходного видео.
- Документация для редактора.

**Выход:** готовность к внутренней эксплуатации.

---

## 10) Роли и процессы
- Редактор:
  - Отбирает новости, проверяет сценарий, запускает рендер.
- Контент-менеджер:
  - Ведет рубрики и резервные новости.
- Тех-оператор:
  - Обновляет ассеты, шаблоны и интеграции моделей.

Процесс выпуска (ежедневный):
1. Сбор новостей автоматически.
2. Редактор подтверждает пул.
3. Генерация сценария и корректировка текста.
4. Preview → правки.
5. Final render и публикация/архив.

---

## 11) Риски и меры
- Риск: низкое качество/шум в источниках.
  - Мера: source scoring + ручной whitelist.
- Риск: долгий рендер.
  - Мера: пресеты качества + отдельный preview-режим.
- Риск: несогласованность озвучки.
  - Мера: строгий phrase bank + кэширование по seed.
- Риск: проблемы с ассетами.
  - Мера: manifest + checksum + валидация перед задачей.

---

## 12) Критерии готовности MVP
- Можно подключить минимум 3 RSS-источника и собрать выпуск из интерфейса.
- Можно выбрать новости, сгенерировать сценарий и отрендерить ролик.
- Работают минимум 3 рубрики (Ursas Index, Top падений, Число дня).
- Фиксированные приветственные фразы звучат одинаково между выпусками.
- Есть логи по каждому шагу пайплайна и статус рендера.
