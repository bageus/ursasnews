# Markdown-план реализации Ursas News (внутренний инструмент)

## 0. Контекст и цель

Нужно реализовать внутреннюю систему, которая объединяет:
1. **Ленту новостей** (RSS + парсеры + AI-поиск/фильтрация).
2. **Программную генерацию видео** по заранее заданному сценарию и готовым ассетам.
3. **Центр рубрик** (Ursas Index, проекты, топ падений, число дня, новости про медведей и fallback-логика).

Ключевые ограничения:
- Интерфейс максимально простой (вкладки, таблицы, формы, без сложного UI).
- Первая версия сцены фиксированная (студия + медведь + доска + fullscreen карточки новости).
- Инструмент внутренний, приоритет — стабильность и скорость сборки выпусков.

---

## 1. Что должно быть на выходе (Definition of Success)

### 1.1 Функциональный результат MVP
- Редактор заходит во вкладку «Лента», получает новости из нескольких источников, фильтрует и выбирает их в выпуск.
- Во вкладке «Генерация ролика» на основе отобранных новостей формируется сценарий и рендерится видео.
- Во вкладке «Рубрики» автоматически/полуавтоматически собираются постоянные секции выпуска.
- Фирменные фразы озвучки звучат одинаково от выпуска к выпуску.

### 1.2 Нефункциональный результат MVP
- Понятный журнал выполнения пайплайна (сбор → AI-обработка → сценарий → рендер).
- Повторяемость результата (одинаковый сценарий + seed + ассеты = одинаковый рендер).
- Управляемость ассетами через манифест и проверку целостности.

## 2. Состав системы

## 2.1 Вкладка A: «Лента новостей»

### Задачи
- Подключить RSS и парсеры.
- Нормализовать данные в единую схему.
- Удалять дубли.
- Давать редактору быстрый интерфейс фильтрации и отбора.
- Добавить AI-поиск/AI-суммаризацию.

### Что нужно получить
- Конфиг источников (`sources.yaml`/`sources.json`).
- Таблица новостей с полями:
  - `title`, `summary`, `source`, `published_at`, `url`, `topic_tags`, `is_duplicate`, `score`.
- Фильтры:
  - по источнику, дате, теме, уровню «медвежести»/тональности.
- Маркеры:
  - `рынок`, `крипта`, `акции`, `ursas ecosystem`, `bear-world`.
- AI-поиск:
  - поле запроса + кнопка «подобрать новости».
- Кнопка «Добавить в выпуск» и очередь выбранных новостей.

### Пути решения
- **Путь 1 (быстрее для MVP):** только RSS + 1-2 простых парсера по CSS-селекторам.
- **Путь 2 (масштабируемый):** adapter framework (отдельный адаптер на источник) + unit-тесты адаптеров.
- **Рекомендация:** стартовать с Пути 1, затем расширить до Пути 2.

---

## 2.2 Вкладка B: «Генерация ролика»

### Задачи
- Дать редактору контроль над сценарием.
- Автоматизировать рендер фиксированной сцены.
- Поддержать предпросмотр и финальный экспорт.

### Что нужно получить
- Формат сценария `episode_script.json`:
  - `intro`, `news_segments[]`, `rubrics[]`, `outro`.
  - таймкоды, текст, настройки субтитров, переходы.
- UI-блоки:
  - список сегментов,
  - редактирование текста,
  - переключатель субтитров,
  - параметры рендера (разрешение/FPS/формат).
- Рендер-кнопки:
  - `Preview` (быстро, сниженное качество),
  - `Final Render` (боевое качество).
- Отображение статуса задачи рендера (Queued/Running/Done/Failed).

### Пути решения
- **Путь 1 (MVP):** ffmpeg-композиция на основе слоёв и таймлайна.
- **Путь 2:** real-time движок (сложнее и дольше).
- **Рекомендация:** путь 1 для MVP как самый предсказуемый и дешевый по времени.

---

## 2.3 Вкладка C: «Рубрики»

### Задачи
- Централизовать постоянные сегменты выпуска.
- Настроить автозаполнение и fallback-логику.

### Что нужно получить
- Панель рубрик со статусом данных за сегодня:
  - Ursas Index,
  - Ursas Projects,
  - Top-10 падающих,
  - Число дня,
  - Ursas Tube,
  - Новости из мира обычных медведей.
- Правила fallback:
  - если нет новой bear-world новости за день → взять из пула `fallback_bear_news`.
  - использованную fallback-новость пометить `used=true`, повторно не выдавать.

### Пути решения
- **Путь 1:** расчет рубрик по cron-задачам (каждый день/каждый час).
- **Путь 2:** расчет on-demand при открытии вкладки.
- **Рекомендация:** комбинированно (cron + ручной пересчет кнопкой).

## 3. Видео-сцена MVP (детализация)

## 3.1 Фиксированная постановка
- Студия.
- Медведь за столом.
- Анимируемые каналы: рот, руки (редко), поворот головы (легко).
- Фон: доска с заголовком новости.
- Переход: доска → fullscreen карточка.
- Опциональные субтитры внизу.

## 3.2 Минимальный таймлайн сегмента новости
1. `0-1.5с`: появление заголовка на доске.
2. `1.5-8с`: дикторская реплика (рев-озвучка + lip-sync).
3. `8-12с`: fullscreen карточка новости.
4. `12-14с`: короткий комментарий.
5. `14-15с`: возврат к студии.

## 3.3 Что должно быть готово по ассетам
- Спрайт/слои медведя (neutral, mouth_open variants, hand_move variants).
- Фон студии.
- Шаблон доски и fullscreen-карточки.
- Заставки рубрик (минимум статические).
- Базовый шрифт и стиль субтитров.

---

## 4. Озвучка «язык рева»

## 4.1 Цель
Сделать узнаваемую озвучку, где ключевые фразы звучат стабильно и одинаково.

## 4.2 Что нужно получить
- `phrase_bank.json`:
  - список фиксированных фраз,
  - привязанный аудио-шаблон/seed.
- Генератор для новых фраз:
  - текст -> последовательность «рев-токенов» -> аудио.
- Кэш:
  - ключ = hash(text + voice_profile + speed + pitch).

## 4.3 Пути решения
- **Путь 1:** предварительно записанные шаблоны + склейка (быстро, но ограничено).
- **Путь 2:** параметрический синтез «рева» (гибко, сложнее).
- **Рекомендация:** гибрид — фикс-фразы из банка + генерация для остального.

---

## 5. Ассеты и хранение

## 5.1 Что нужно получить
- `asset_manifest.json`:
  - `asset_id`, `category`, `version`, `uri`, `checksum`, `updated_at`, `active`.
- Проверка ассетов перед рендером:
  - наличие файла,
  - checksum,
  - совместимость версий.

## 5.2 Пути хранения
- **Вариант A:** всё в репозитории (быстро стартовать, но тяжело масштабировать).
- **Вариант B:** гибрид (репо + external storage).
- **Рекомендация:** вариант B:
  - в репо только конфиги/манифесты,
  - тяжелые ассеты — в NAS/S3/Minio.

## 6. Архитектура и компоненты

## 6.1 Frontend (внутренний кабинет)
- Простое SPA с 4 вкладками:
  - `FeedDashboard`,
  - `VideoBuilder`,
  - `RubricsCenter`,
  - `Settings`.

## 6.2 Backend API
- `POST /ingest/run`
- `GET /news`
- `POST /news/select`
- `POST /script/generate`
- `PUT /script/{id}`
- `POST /render/preview`
- `POST /render/final`
- `GET /render/{id}/status`
- `GET /rubrics/today`
- `POST /rubrics/recalculate`

## 6.3 Workers
- `collector-worker` — сбор RSS/парсеров.
- `nlp-worker` — AI-классификация/суммаризация.
- `voice-worker` — генерация рев-озвучки.
- `render-worker` — ffmpeg-рендер.

## 6.4 Данные
- PostgreSQL — новости, сценарии, рубрики, статусы.
- Redis/Queue — очереди задач.
- Object storage — ассеты и готовые видео.

## 8. Матрица «задача -> результат -> проверка»

| Задача | Что получить | Как проверить |
|---|---|---|
| Подключить RSS | Новости в БД по расписанию | В логах есть успешные ingest-циклы |
| Дедупликация | Нет повторов одной новости | Доля дублей после обработки < заданного порога |
| AI-суммаризация | Короткие тезисы по каждой новости | В UI есть summary и ручная валидация редактором |
| Сценарий | JSON-сценарий выпуска | Проходит schema validation |
| Рендер preview/final | 2 режима видео-сборки | Видео создается и доступно в хранилище |
| Рубрики | Автозаполняемые блоки | Данные появляются ежедневно |
| Fallback bear-news | Нет повторов резервных новостей | Использованная запись помечается и не выдается повторно |
| Phrase bank | Стабильные бренд-фразы | А/Б проверка звучания одинаковых фраз |

## 9. Риски и варианты решения

1. **Шумные/слабые источники новостей**
   - Решение: рейтинг источников + whitelist + ручная модерация.
2. **Долгий рендер**
   - Решение: preview-пресет, очередь задач, ограничение параллелизма.
3. **Сложность в поддержке парсеров**
   - Решение: адаптерная архитектура и health-check на источники.
4. **Нестабильность озвучки**
   - Решение: phrase bank + cache + фиксированные параметры голоса.
5. **Проблемы с ассетами**
   - Решение: manifest + checksum + preflight-валидация.

## 10. Приоритеты запуска (MoSCoW)

### Must
- Лента новостей + ручной отбор.
- Генерация сценария и рендер MVP-видео.
- 3 ключевые рубрики.
- Phrase bank для приветствия/закрытия.

### Should
- AI-поиск и авто-тезисы.
- Fallback новости о медведях без повторов.
- Экспорт в 2-3 формата.

### Could
- Расширенные анимации медведя.
- Более сложные переходы/эффекты.
- Мульти-языковая озвучка.

### Won't (для MVP)
- Свободный видеоредактор уровня NLE.
- Сложная кастомная 3D-сцена.

---

## 11. Чек-лист готовности к внутреннему запуску

- [ ] Вкладка «Лента» стабильно собирает новости минимум из 3 источников.
- [ ] Редактор может сформировать выпуск без ручной подготовки данных вне системы.
- [ ] Вкладка «Генерация ролика» формирует preview и final.
- [ ] Вкладка «Рубрики» обновляет ежедневные показатели.
- [ ] Fallback для bear-world новостей работает без повторов.
- [ ] Ключевые бренд-фразы озвучиваются одинаково.
- [ ] Логи/ошибки доступны и понятны операторам.
- [ ] Есть краткая инструкция для редактора (runbook).

---

## 12. Краткий стартовый roadmap (календарно)

- **Неделя 1:** аналитика, схема данных, источники, шаблон сценария.
- **Неделя 2-3:** лента новостей + AI-поиск + отбор в выпуск.
- **Неделя 4-5:** видео-генерация MVP (preview/final).
- **Неделя 6:** рубрики + fallback.
- **Неделя 7:** voice v1 + стабилизация + документация.

Итого ориентир на MVP: **~7 недель** при небольшой команде.
